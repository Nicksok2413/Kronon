
# 🏗️ Архитектура и ключевые особенности (Kronon OS)

Kronon — это не просто классический монолит на Django. Это **Enterprise-ready** система, спроектированная с упором на асинхронность, целостность данных, безопасность и современный Developer Experience.

В этом документе описаны ключевые архитектурные решения проекта.

---

## 🛡️ 1. Неуязвимый журнал аудита (Audit Log System)

В финансовых и бухгалтерских системах целостность истории изменений критически важна.
Я отказался от стандартного паттерна "переопределения метода `save()`" (сигналов Django) в пользу **строгого аудита на уровне СУБД**.

### Как это работает:
*   **PostgreSQL Triggers (`pghistory` & `pgtrigger`):** Все события (`INSERT`, `UPDATE`, `DELETE`) фиксируются триггерами внутри самой базы данных. Аудит невозможно обойти даже при использовании `bulk_update()`, `QuerySet.delete()` или сырых SQL-запросов (`raw()`).
*   **Инкапсуляция бизнес-контекста (ContextVars):** С помощью кастомного `kronon_history_middleware` (для API) и `TaskWithAudit` (для Celery) пробрасывается контекст выполнения (Correlation ID, ID пользователя, Email пользователя, IP адрес, User-Agent, URL, HTTP-метод, сервис - источник запроса (System, API/Web, Admin, etc), название Celery-задачи, ID Celery-задачи, и команда CLI) прямо в транзакцию БД.
*   **Денормализация для скорости:** Весь контекст сохраняется в `JSONB` колонке (`pgh_context`) рядом с событием. Это избавляет от тяжелых `JOIN` запросов при построении истории (миллионы строк читаются за миллисекунды).
*   **Иммутабельность (Append-Only):** Таблицы истории защищены триггером `pgtrigger.AppendOnly`. Никто (даже администратор БД) не может изменить или удалить записи из журнала аудита без жесткого вмешательства в структуру СУБД.
*   **Smart Diffs:** База данных сама вычисляет разницу изменений (`pgh_diff`) в формате `{"field": ["old", "new"]}`, снимая вычислительную нагрузку с Python-воркеров.

## ⚡ 2. Асинхронность и производительность (High Performance)

Проект использует возможности Python 3.14 и PostgreSQL 18.1 по максимуму.

*   **100% Async API:** Используется `Django Ninja` (поверх Starlette) с асинхронными роутерами, сервисами и селекторами.
*   **Async ORM & Psycopg 3:** Взаимодействие с базой данных не блокирует Event Loop. Используются `afirst()`, `acreate()`, `asave()` и асинхронные генераторы (`async for`).
*   **AIO в PostgreSQL 18:** База данных настроена на использование Asynchronous I/O (`effective_io_concurrency=200`), что радикально ускоряет seq scans и работу с индексами.
*   **Transaction Pooling:** Приложение подключается к БД через **PgBouncer** в режиме `transaction`. Для предотвращения конфликтов Django автоматически отключает `SERVER_SIDE_CURSORS` при детекте порта пулера (`6432`).

## 🔐 3. Безопасность и контроль доступа (Security, RBAC & OLP)

Разделены аутентификация, ролевая модель (RBAC) и объектные права (OLP), смещая тяжелые проверки на уровень SQL.

*   **Push-down OLP:** Объектные права (Object-Level Permissions) вычисляются не в Python цикле, а на уровне `QuerySet`. Метод `SoftDeleteQuerySet.for_user()` динамически применяет `models.Q` фильтры. Линейный бухгалтер физически не может выгрузить из базы чужого клиента.
*   **Hybrid JWT & API Key Auth:** Система поддерживает как JWT-токены (с `Blacklist` ротацией) для пользователей, так и статические API-ключи для межсервисного взаимодействия, резолвя их в единый интерфейс `get_auth_identity`.
*   **Force Logout:** Мгновенное (Soft Delete) отключение пользователей прерывает их сессии на уровне Middleware, блокируя любые дальнейшие действия даже с валидным JWT-токеном.

## 🐘 4. Database Mastery

*   **Native UUIDv7:** В качестве Primary Keys используются `UUIDv7` (нативная фича Python 3.14 / Postgres 18). Это решает проблему фрагментации B-Tree индексов, свойственную UUIDv4, и позволяет базе сортировать записи по дате создания (`ORDER BY -id`) без дополнительных индексов на `created_at`.
*   **Soft Delete:** Встроенная поддержка мягкого удаления через кастомные `SoftDeleteManager` и `SoftDeleteQuerySet`. Записи помечаются `deleted_at=NOW()` атомарно, сохраняя ссылочную целостность (Foreign Keys) и журнал аудита.
*   **GIN & Trigram Search:** Реализован полнотекстовый нечеткий поиск (Fuzzy Search) по текстовым полям (названия, УНП) через `pg_trgm` и `GinIndex`. Это обеспечивает молниеносный автокомплит в API и Админке.

## 🛠️ 5. Developer Experience (DX) & CI/CD

*   **Strict Typing (Mypy):** Проект использует 100% строгую типизацию (`disallow_any_generics = true`). Для интеграции дженериков с рантаймом Django применяется патч `django-stubs-ext`.
*   **Pydantic V2 Value Objects:** Сложные JSON-поля в БД (например, контакты) типизируются и валидируются "на лету" через Pydantic-модели (`get/set` property методы), обеспечивая строгую структуру NoSQL данных внутри SQL.
*   **Lightning Fast Tests (RAM Disk):** В CI/CD и локальном `docker-compose.test.yml` PostgreSQL разворачивается в оперативной памяти (`tmpfs`) с отключенным `fsync`. В связке с `pytest-xdist` это ускоряет тесты в 5-10 раз.
*   **Safe Multi-stage Docker:** Docker-образы собираются по Multi-stage паттерну. Приложение запускается от `non-root` пользователя (`kronon`), а скрипты инициализации используют утилиту `su-exec` для безопасного понижения привилегий.
*   **Make Automation:** Вся рутина скрыта за `Makefile`. Команды вроде `make check-all` или `make reset-migrations` позволяют управлять циклом разработки одной рукой.

## 📊 6. Observability (Наблюдаемость)

*   **Distributed Tracing (Correlation ID):** Каждому запросу к API, задаче Celery или команде CLI присваивается уникальный `X-Correlation-ID`. Этот ID пробрасывается через все слои (Loguru -> СУБД (Audit Log) -> Sentry). Вы можете связать ошибку в Sentry с конкретной записью в БД и логом воркера.
*   **Sentry Integration:** Интеграция с Sentry настроена на автоматический перехват 500-х ошибок, логирование performance-метрик и захват локальных переменных.
*   **Metrics Ready:** Инфраструктура готова к подключению Prometheus + Grafana (экспортеры Celery и БД прописаны в Compose-файлах).

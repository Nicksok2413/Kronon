###  Архитектурные особенности проекта

---

#### 🏆 Журнал аудита (Audit Log)

Audit Log реализован с помощью `pghistory` и `pgtrigger`.
Сами физические таблицы (`clients_clientevent, users_userevent, etc`) и триггеры находятся в моделях соответствующих приложений (декоратор `@pghistory.track`),
что сохраняет целостность БД, но вся логика отображения, админка (`KrononEventsAdmin`), API (`apps/audit/api/v1.py`) и кастомный `Middleware` вынесена в `apps/audit`.

Реализована Proxy-модель `KrononEvents` — это паттерн, который советуют использовать авторы pghistory для кастомизации.


`kronon_history_middleware` (для API) и `TaskWithAudit` (для Celery) пробрасывают контекст выполнения (Correlation ID, ID пользователя, Email пользователя, IP адрес, User-Agent, URL, HTTP-метод, сервис - источник запроса (System, API/Web, Admin, etc), название Celery-задачи, ID Celery-задачи, и команда CLI).

`kronon_history_middleware` реализован в виде фабрики гибридного middleware по стандартам Django 6.0.
Использует `@sync_and_async_middleware` для гарантии сохранности `ContextVars` в гибридном ASGI/WSGI окружении.

Утилита `aexecute_with_audit` оборачивает синхронные методы (Client.objects.create, client.save, etc) в синхронную функцию (_wrapper) с менеджером контекста `pghistory.context` (основанном на ContextVars),
а затем запускает её в пуле потоков (Thread Pool) через sync_to_async, не блокируя Event Loop и гарантируя сохранность контекста — это самый безопасный и надежный паттерн конкурентности для гибридных фреймворков.

#### 🏆 OLP (Объектные права) на уровне БД
Многие разработчики тащат все записи из БД, а потом фильтруют их через циклы, генераторы или list comprehensions в Python (`for client in clients`), что убивает память и вызывает проблему N+1.

Мой подход генерирует оптимальный SQL-запрос `WHERE accountant_id = X OR primary_accountant_id = X OR ...` прямо на уровне БД.
Это работает за миллисекунды даже на миллионах записей.

В `apps/common/managers.py` реализован `SoftDeleteQuerySet` - кастомный QuerySet,
реализующий логику мягкого удаления (Soft Delete) и нативную OLP-фильтрацию прав доступа на уровне БД (`метод for_user()`).
Каждая модель, где нужна OLP-фильтрация (Client, Contract, etc) имеет метод `get_olp_filter`, который возвращает Q-объект (`models.Q`).

#### 🏆 Распределенная трассировка (Distributed Tracing)
Обычно микросервисы и фоновые задачи — это "черная дыра" для логов.

В проекте реализована распределенная трассировка через `correlation_id`.
Этот ID пробрасывается от Sentry и Loguru до самых воркеров Celery.
Любой запрос, от API до фонового письма, можно отследить по одному заголовку `X-Correlation-ID`.

В `config/celery.py` использован декоратор `before_task_publish` для прокидывания `correlation_id` в заголовки брокера (RabbitMQ/Redis) и последующее его извлечение в воркере.

---

#### 🏆 Тестовая инфраструктура (RAM Disk + xdist)
Использование `tmpfs: /var/lib/postgresql` (RAM Disk) и отключение `fsync`, `synchronous_commit` и `full_page_writes` в `docker-compose.test.yml` в связке с `pytest -n auto --dist loadscope` (распараллеливание тестов по ядрам) дает фантастическую скорость.
Пока тестов мало, разница незаметна, но когда их будет 1000+, они будут проходить за секунды, а не минуты.

---

#### 🏆 Безопасность (Force Logout)
Проверка `is_deleted` в `kronon_history_middleware` и вызов `alogout(request)`.
Если админ уволил сотрудника (Soft Delete), его текущие JWT-сессии или сессии админки инвалидируются на лету при следующем запросе.

---

#### 🏆 Единая система ошибок (ErrorOut)
Внедрены STANDARD_ERRORS и перехватчики в apps/common/exceptions.py.
Swagger выглядит опрятно, формат 400/403/404/422 ошибок всегда предсказуем.

#### 2. Нюанс с `pytest-xdist` и БД
В `Makefile` тесты запускаются так: `pytest -n auto --dist loadscope`.
Плагин `pytest-django` умеет работать с `xdist`. Он создает отдельные базы данных для каждого воркера (например, `test_kronon_gw0`, `test_kronon_gw1`).
Однако, в `pyproject.toml` жестко захардкожено имя тестовой базы:
```toml
"DATABASE_URL=postgres://postgres:postgres@localhost:5433/postgres",
```
Если все 4-8 воркеров xdist пойдут в одну базу `postgres` писать тестовые данные, они могут передраться (Deadlocks, Transaction errors).
**Решение:** Оставить URL без имени базы (или указать дефолтную), а `pytest-django` сам подставит суффиксы для воркеров, если не блокировать его логику.
Проследить за этим: если при увеличении числа тестов они начнут "мигать" (flaky tests) и падать с ошибкой БД — нужно будет разрешить `pytest-django` создавать БД динамически (убрать `DATABASE_URL` из `pytest-env` и прокинуть параметры БД через стандартные настройки Django для тестов).

#### 3. Достроить фундамент бизнес-логики
Без **Пользователей (Сотрудников)** и **Договоров (Contracts)** система не имеет смысла.
Мы не сможем биллить и не сможем назначать задачи (Tasks).

План:

**Опция А: Модуль Users & Departments (API + Services)**
Уже есть модель User, но нет API для управления отделом кадров (HR).
*   CRUD для пользователей (прием на работу, увольнение).
*   CRUD для отделов.
*   Назначение ролей.

**Опция Б: Контракты и Услуги (Contracts & Services)**
*   Модель контракта с клиентом (Дата начала, дата конца, статус).
*   Тарифная сетка (Ежемесячный платеж, включенные операции).
*   Связь клиента с договорами (У одного клиента может быть история договоров).

**Опция В: Документы (Хранилище)**
*   Интеграция с AWS S3 (или MinIO локально).
*   API для безопасной загрузки сканов паспортов, УНП и договоров.
*   Привязка файлов к Клиентам.
